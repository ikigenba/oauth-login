# Decision 4 тАФ The loopback callback listener

## Decision.

`internal/callback` binds the loopback interface, serves one route, and returns
the first valid redirect.

```go
type Server struct {
    Port         int    // 0 тЖТ OS-assigned ephemeral
    CallbackPath string // e.g. "/auth/callback"
    State        string // expected; mismatch is fatal
}

// Listen binds before the browser is opened, so the port is known
// and a bind failure is reported before the user is sent anywhere.
func (s *Server) Listen() (port int, err error)

// Wait serves until the first valid callback, ctx expiry, or a fatal callback.
func (s *Server) Wait(ctx context.Context) (Result, error)
```

**Binding.** `127.0.0.1` on the requested port; with `Port == 0` the OS assigns
one and `Listen` reports it back, so the redirect URI can be composed from the
real port. The same port is then bound on `[::1]`. **An IPv6 bind failure is
non-fatal** тАФ logged to stderr, and the server proceeds IPv4-only. Both
listeners feed one handler and one result channel. Binding happens in `Listen`,
before the browser opens, so "port already in use" surfaces immediately rather
than after the user has authenticated.

**Routing.** Only `CallbackPath` is the callback. **Any other path gets 404 and
does not end the flow** тАФ browsers request `/favicon.ico` unprompted, and
treating the first request as the callback would end the login on a favicon
fetch.

**Handling, in order.** On a request to `CallbackPath`:

1. `state` absent or not equal to `State` тЖТ error page, fatal, and the code is
   never exchanged. Checked first, before anything in the query is trusted.
2. `error` present тЖТ error page showing `error` and `error_description`, fatal.
3. `code` absent (and no `error`) тЖТ error page, fatal. It arrived at our path,
   so it is malformed, not stray.
4. Otherwise тЖТ success page ("Login complete тАФ return to your terminal"),
   `Result{Code: ...}` returned. The page is a minimal inline HTML document
   with no external references, so it renders offline and phones nowhere.

**First valid callback wins.** The result is delivered once, via a
`sync.Once`-guarded send, and the listeners stop accepting. A duplicate
arriving during shutdown gets whatever the closing server gives it and cannot
change the outcome. The port is not held open to serve a nicer second page.

**Deadline.** `Wait` honours `ctx`, whose deadline the caller sets from
`--timeout`. It covers waiting on the human only: it starts when `Wait` is
called and **stops the moment a valid callback is in hand**, so the token
exchange runs under its own HTTP timeout instead. Expiry returns an error
naming the elapsed budget.

## Rejected.

- **Binding `localhost` by name.** Resolution order decides which family we
  land on, so the listener could end up somewhere the browser will not look.
  Binding the literals and covering both families removes the ambiguity, as
  RFC 8252 Section 7.3 advises.
- **Making an IPv6 bind failure fatal.** Would fail logins on hosts with no
  IPv6 stack, for a listener that is redundant whenever IPv4 works.
- **Treating the first request on any path as the callback.** A real failure
  mode, not hypothetical: an unprompted favicon fetch would end the login.
- **Ignoring a callback with neither `code` nor `error`.** It arrived at our
  path; silently waiting for another leaves the user staring at a browser that
  believes it finished, until the timeout.
- **Checking `state` after reading `code`.** Reverses the security order тАФ the
  point of `state` is to decide whether the rest of the query is ours at all.
- **One deadline across the whole flow.** A user who took 4m59s would see a
  slow token exchange reported as a timeout, which is the most confusing
  possible message for the user already having the worst time.

## Verification.

- R-11L3-FGOF тАФ With `Port == 0`, `Listen` binds `127.0.0.1` on an
  OS-assigned port and returns that non-zero port; a TCP connection to it on
  `127.0.0.1` succeeds.
- R-12SZ-T8F4 тАФ After a successful `Listen`, a request to `[::1]` on the same
  port reaches the same handler, on a host where IPv6 loopback is available.
- R-140W-705T тАФ When the IPv6 bind fails, `Listen` returns successfully with
  the IPv4 listener serving, and a callback delivered over IPv4 completes the
  flow.
- R-158S-KRWI тАФ With an explicit non-zero `Port`, the listener binds exactly
  that port and `Listen` returns it; a port already in use returns an error
  from `Listen` rather than from `Wait`.
- R-16GO-YJN7 тАФ A request to `/favicon.ico` returns 404 and `Wait` does not
  return; a subsequent valid callback still completes it.
- R-18WH-Q34L тАФ A request to `CallbackPath` with matching `state` and a `code`
  returns `Result` carrying exactly that code, and responds 200 with an HTML
  body containing no external resource references.
- R-1A4E-3UVA тАФ A callback whose `state` differs from `State`, and one with no
  `state` at all, each return an error and no code, and the response is an
  error page rather than the success page.
- R-1BCA-HMLZ тАФ A callback carrying `error=access_denied` and an
  `error_description` returns an error whose text contains both, even when a
  `code` is also present.
- R-1CK6-VECO тАФ A callback with matching `state` but neither `code` nor
  `error` returns an error rather than continuing to wait.
- R-1DS3-963D тАФ After the first valid callback, `Wait` has returned and the
  listener no longer accepts new connections on the bound port; a second
  callback cannot produce a second `Result`.
- R-1EZZ-MXU2 тАФ With a context whose deadline expires before any callback,
  `Wait` returns an error naming the configured budget, and no code.
- R-1G7W-0PKR тАФ A valid callback arriving before the deadline returns its
  result, and the deadline expiring afterwards neither cancels the result nor
  produces an error.

These run against the real listener over real loopback TCP with a real
`net/http` client standing in for the browser, so a routing, binding, or
shutdown mistake fails rather than being accepted by a stub.
