# Decision 6 — CLI surface, validation, and the composition root

## Decision.

`cmd/oauth-login` owns the flag surface, validates it fully before anything
observable happens, wires the seams, and maps outcomes to streams and exit
codes.

**Flags**, parsed with the standard library `flag` package:

| flag | default | meaning |
|---|---|---|
| `--auth-url` | *required* | authorization endpoint |
| `--token-url` | *required* | token endpoint |
| `--client-id` | *required* | OAuth client id |
| `--scope` | *(none)* | space-separated scopes; omitted entirely when unset |
| `--client-secret` | *(none)* | confidential client secret, sent in the form body |
| `--callback-host` | `localhost` | host used **in the redirect URI string** |
| `--port` | `0` | callback port; `0` takes an OS-assigned ephemeral port |
| `--callback-path` | `/callback` | callback route, and the redirect URI path |
| `--auth-param` | *(repeatable)* | extra `key=value` on the authorize URL |
| `--token-param` | *(repeatable)* | extra `key=value` in the token request body |
| `--token-header` | *(repeatable)* | extra `key=value` header on the token request |
| `--no-browser` | `false` | do not launch a browser; still print the URL |
| `--timeout` | `5m` | how long to wait for the callback |

`--callback-host` names the string providers match against; the listener always
binds the loopback literals (D4). The default is `localhost` because that is
the widely registered form; users whose provider registered the IP literal pass
`--callback-host 127.0.0.1`.

**Redirect URI** is composed as `http://<callback-host>:<port><callback-path>`,
using the port `Listen` actually bound. With `--port 0` this means binding
precedes URL construction, since the ephemeral port is not known until then.

**Validation, all before the browser opens.** Missing required flag; a
repeatable flag value with no `=`; an `--auth-param` or `--token-param`
colliding with a reserved key (D2, D3); and `--client-secret` supplied together
with an `Authorization` header from `--token-header`, which RFC 6749 forbids as
two authentication methods in one request. Each is a startup error naming the
offending flag, exit 1, with nothing spawned and no URL opened.

**Order of operations**, chosen so nothing user-visible happens before failure
is ruled out: parse and validate → bind the listener → compose redirect URI →
build authorize URL → print it to stderr → launch browser → wait → exchange →
write bytes to stdout → exit 0.

**Streams and exit.** On success, the token response bytes go to stdout in a
single write and nothing else ever does. Every diagnostic goes to stderr. Any
failure exits 1 having written nothing to stdout.

**`--help`** describes every flag and ends with a worked example that runs:

```
oauth-login \
  --auth-url  https://auth.openai.com/oauth/authorize \
  --token-url https://auth.openai.com/oauth/token \
  --client-id app_EMoamEEZ73f0CkXaXp7hrann \
  --scope "openid profile email offline_access" \
  --port 1455 --callback-path /auth/callback \
  > auth.json
```

The example needs no `--callback-host` because the default already matches
OpenAI's registered `http://localhost:1455/auth/callback`, and the fixed port
and path are exactly why those two flags exist. The help also shows the Basic
authentication form, since assembling it is the fiddly part:

```
--token-header "Authorization=Basic $(printf '%s:%s' "$ID" "$SECRET" | base64 -w0)"
```

This example is the only place in the binary naming a real provider, and it is
documentation, not behavior.

## Rejected.

- **A richer exit-code taxonomy.** The failure modes do not group into
  distinctions a caller could act on, and inventing codes locks us into them.
  Widening later is compatible; narrowing is not.
- **`--client-auth basic|post`.** Encodes a taxonomy of provider behavior,
  which is provider knowledge smuggled into a tool that claims none.
  `--token-header` is mechanism instead, and composes with any other header a
  provider demands.
- **Body-only client authentication.** RFC 6749 makes Basic the one method
  servers must support, so a Basic-only provider would have had no workaround.
- **Letting `--auth-param redirect_uri=...` override the redirect.** It would
  disagree with the address actually bound, producing a login that hangs until
  the timeout — the worst failure shape available.
- **Defaulting `--callback-host` to `127.0.0.1`.** What RFC 8252 prefers, but
  it fails out of the box against the common registered form. The RFC's
  concerns are met by binding literals and covering both families instead.
- **Opening the browser before binding.** Sends the user to authenticate
  against a listener that may not exist.
- **A `--quiet` or `--verbose` flag.** Stdout is already machine-clean by
  construction, so the volume on stderr is a human's concern and small.

## Verification.

- R-1NJA-BC0X — Omitting `--auth-url`, `--token-url`, or `--client-id` exits 1
  with a message on stderr naming the missing flag, having written nothing to
  stdout.
- R-1PZ3-2VIB — `--client-secret` together with a `--token-header` whose key is
  `Authorization` (in any letter case) exits 1 naming both flags, before any
  network activity; either alone is accepted.
- R-1R6Z-GN90 — A reserved-key collision on `--auth-param` or `--token-param`,
  and a repeatable flag value lacking `=`, each exit 1 before the listener
  binds and before the launcher is invoked.
- R-1SEV-UEZP — On a successful login, stdout holds exactly the token
  endpoint's response bytes and nothing else: no trailing newline is added, no
  progress text is interleaved, and the bytes match the provider's body
  exactly.
- R-1TMS-86QE — On each failure mode (validation error, provider `error=`,
  `state` mismatch, non-2xx token response, timeout), stdout receives zero
  bytes.
- R-1UUO-LYH3 — The authorize URL, progress, and error text appear on stderr,
  including when stdout is redirected to a file, and none of them appear in
  that file.
- R-1W2K-ZQ7S — A completed login exits 0; every failure mode above exits
  exactly 1, never 2 or any other non-zero value.
- R-1XAH-DHYH — `--help` exits 0 and its output names every flag in the table
  above, and contains the worked example's authorize URL, token URL, client
  id, `--port 1455`, and `--callback-path /auth/callback`.
- R-1YID-R9P6 — With no `--callback-host`, the redirect URI sent to the
  provider and the one used at exchange both use host `localhost`; with
  `--callback-host 127.0.0.1` both use `127.0.0.1`, and the two values are
  always identical to each other.
- R-1ZQA-51FV — With no `--timeout`, the callback deadline is 5 minutes; an
  explicit `--timeout` is honoured, and expiry exits 1.
- R-20Y6-IT6K — The redirect URI is `http://<host>:<port><path>` built from the
  port actually bound: with `--port 0` it carries the OS-assigned port rather
  than `0`, and with `--port 1455 --callback-path /auth/callback` it is exactly
  `http://localhost:1455/auth/callback`.

These drive the built binary end to end against an `httptest.Server` acting as
a validating provider — one that redirects to the real listener and checks the
exchange — with a fake launcher standing in for the browser. Stdout and stderr
are captured separately so the stream discipline is observed rather than
assumed. The claim that a real provider accepts all this is D7's.
