# Decision 8 ‚Äî Build and install tooling

## Decision.

A root `Makefile` mirroring `agentrepl`'s, so a person moving between the
sibling repositories finds the same target names doing the same things.

```make
BINARY  := oauth-login
BIN_DIR := bin
PREFIX  ?= $(HOME)/.local

.PHONY: build fmt test check install clean

build: $(BIN_DIR)/$(BINARY)

$(BIN_DIR)/$(BINARY): go.mod $(wildcard go.sum) $(shell find cmd internal -name '*.go')
	install -d $(BIN_DIR)
	go build -o $(BIN_DIR)/$(BINARY) ./cmd/oauth-login

fmt:
	go fmt ./...

test:
	go test ./...

check:
	go build ./...
	go vet ./...
	go test ./...
	@test -z "$$(gofmt -l .)" || { gofmt -l .; exit 1; }

install: build
	install -d $(PREFIX)/bin
	install -m 0755 $(BIN_DIR)/$(BINARY) $(PREFIX)/bin/$(BINARY)

clean:
	rm -rf $(BIN_DIR)
	go clean
```

`build` is a real file target, not phony, so a rebuild is skipped when nothing
changed. `bin/` is build output and is not tracked.

**Two deliberate departures from the mirrored original**, both preserving the
target names and their meanings:

- **`go.sum` is wrapped in `$(wildcard ...)`.** This module has no dependencies
  and therefore no `go.sum`; listing it directly makes `make build` fail with
  `No rule to make target 'go.sum'`. The wildcard makes the file participate
  when it exists and be absent without error until then, so nobody has to
  remember to restore the prerequisite when the first dependency arrives.
- **`check` is added.** The mirrored `fmt` target runs `go fmt ./...`, which
  *rewrites* files rather than checking them, so nothing in the original
  asserts the green bar. `check` runs exactly the four commands the design
  Conventions define as green, with the `gofmt` step failing when any file
  would be reformatted. `fmt` keeps its original rewrite behavior.

`check` is a convenience wrapper, not a redefinition: the Conventions remain
the authority on what green means, and `check` must be kept equal to them.

## Rejected.

- **A byte-faithful mirror.** `make build`, the primary target, would fail on a
  fresh checkout because of the missing `go.sum`, and the repository would
  have no target that checks the green bar. A mirror whose main target does
  not run is not worth the fidelity.
- **Renaming targets to something more descriptive** (`all`, `verify`,
  `lint`). Fidelity to the sibling is the entire point; a developer who runs
  `make build test install` in one repo must get the same in the other.
- **Making `fmt` a check rather than a rewrite.** Would silently change the
  meaning of a mirrored target ‚Äî someone expecting their files formatted would
  get a failure instead. `check` carries the gate; `fmt` keeps its behavior.
- **Deleting `go.sum` from the prerequisites entirely.** Correct today, wrong
  the moment a dependency is added, and wrong silently: the binary would stop
  rebuilding when only the dependency set changed.
- **Adding targets with no counterpart in the sibling** (release, tag, cross
  compile). Versioning is annotated git tags cut by hand; automating it here
  invents a workflow neither repository has.

## Verification.

Structural ‚Äî this Decision produces build tooling, not program behavior. It
carries **no ids**, deliberately: its proof is running the real targets and
observing the real artifacts, which is exactly the deterministic exit
condition its phase carries. Wrapping `make` in a Go test would add a slow,
environment-dependent test whose substrate is weaker than simply invoking the
targets, and the binary's own behavior is already covered by D2 through D7.
