# Decision 7 — Live verification against a real provider

## Decision.

Every other Decision is proven against an `httptest.Server` we wrote. That
substrate is real HTTP over real loopback, and it *validates* rather than
accepts, so it can falsify us — but it can only falsify us against our own
reading of the specification. The claim it structurally cannot test is the one
that matters most: **that a real authorization server accepts what we send and
returns tokens.**

So one test drives the real thing.

**Placement.** `cmd/oauth-login/live_test.go`, behind `//go:build live`. It is
excluded from `go test ./...` and therefore from the green bar. A human runs
it deliberately:

```
go test -tags live ./cmd/oauth-login/ -run TestLiveLogin -v
```

**What it does.** Builds and runs the binary with the flags from the `--help`
example, against real endpoints, requiring a human to complete a browser login,
and asserts on the **observable outcome**: exit 0, stdout parses as JSON, and
that JSON carries a non-empty `access_token`. Parsing happens in the *test*,
never in the binary — the test is allowed knowledge of the provider that the
binary is not.

Configuration comes from the environment (endpoints, client id, port and path)
so no provider constant is compiled into the tree; the test skips with a clear
message when they are unset, rather than failing.

**What "done" means for this Decision, stated plainly.** The spec's bar is that
the test **exists, compiles under its tag, and its procedure is documented**.
Whether a human has actually run it against a live account is outside what the
spec can mechanically check, and the build loop is never blocked on it. An
unrun live smoke is therefore **not** a coverage gap and must not be treated as
one. Running it is a human task, performed when the binary is released or when
a provider's behavior is in doubt.

## Rejected.

- **No live test at all.** Leaves "a real provider accepts our request"
  entirely unproven, with our own fake as the only judge of our correctness —
  the exact circular verification the design guidance warns about.
- **Running it in the default suite.** It needs a human at a browser and real
  credentials; in `go test ./...` it would be permanently red or permanently
  skipped, and a permanently skipped test teaches everyone to ignore skips.
- **Asserting only that the process exited 0.** Exit 0 with empty stdout would
  pass. The proof has to be a token that came back.
- **Compiling the provider constants into the test.** Would put a real
  provider's identity in the tree as behavior rather than documentation, which
  is the provider-neutrality rule this design's Conventions establish.
- **Automating the browser step.** Scripting a real login is brittle,
  provider-specific, and a standing invitation to break terms of service.

## Verification.

- R-2262-WKX9 — A live login against a real, protocol-compliant authorization
  server, driven by the built binary through a human browser sign-in, exits 0
  and writes a token response to stdout containing a non-empty `access_token`.
  Substrate: the real provider over the network, not a fake. Realized by
  `cmd/oauth-login/live_test.go` under `//go:build live`; the phase's bar is
  that this test exists, compiles under its tag, and is documented, since its
  execution requires a human and is deliberately outside the suite.
