# Decision 3 — Token exchange and verbatim passthrough

## Decision.

`internal/oauth` exchanges the authorization code and returns the response
body as **raw bytes it never parses**.

```go
func (c *Client) Exchange(
    ctx context.Context,
    s *Session,
    code string,
    extra []Param,
    headers []Param,
) ([]byte, error)
```

**Request.** `POST` to `TokenURL`, content type
`application/x-www-form-urlencoded`, carrying `grant_type=authorization_code`,
`code`, `code_verifier`, `redirect_uri`, and `client_id`. `client_secret` is
added when configured. `extra` (`--token-param`) is appended; `headers`
(`--token-header`) is applied to the request.

**Reserved-key rejection**, matching D2: `grant_type`, `code`,
`code_verifier`, `redirect_uri`, `client_id`, and `client_secret` are ours. A
colliding `--token-param` is an error before any request is sent.

**Success.** On a 2xx, the response body is read whole and returned
**unmodified** — the exact bytes the provider sent, with key order, whitespace,
and formatting intact. The package never unmarshals it, never re-encodes it,
never validates that it is JSON, and never inspects it for an `error` key.
This is not laziness: parsing and re-emitting would silently normalize the
caller's file, and interpreting the payload is precisely the provider-specific
knowledge this tool refuses to hold.

**Failure.** On a non-2xx, `Exchange` returns an error carrying the status and
the response body text (truncated for display), and returns **no bytes**. The
body reaches the user on stderr via the composition root; it never reaches
stdout. A transport-level failure returns an error the same way.

The `[]byte`-or-`error` signature is what makes the stream discipline
structural: there is no return shape in which a caller holds both a payload
and a failure, so "wrote an error body to stdout" is not an available mistake.

## Rejected.

- **Returning a parsed struct or `map[string]any`.** Any parse-and-re-emit
  round trip reorders keys and reformats whitespace, breaking the promise that
  the file is what the provider said. It would also force us to have opinions
  about fields, which is where provider knowledge creeps in.
- **Unconditional passthrough regardless of status.** More literally faithful
  to "verbatim", but it writes an `{"error":...}` object into
  `~/.foo/auth.json`, producing a file shaped like credentials that is not
  credentials. Every consumer would then have to sniff for an `error` key
  before trusting the file, which is exactly the burden the exit code exists to
  remove.
- **Validating that the body is JSON before returning it.** Rejects a
  conforming-but-unusual provider for no benefit, and asserts an opinion about
  the payload we promised not to hold.
- **`--token-header` applied to the authorize step too.** The authorize step is
  a browser navigation, not a request we issue; there is nothing to attach a
  header to.
- **Streaming the body straight to stdout.** Would emit partial output before
  the status is known, making a mid-transfer failure leave a truncated file.

## Verification.

- R-0QLZ-ZJ06 — The exchange request is a `POST` with content type
  `application/x-www-form-urlencoded` whose body carries exactly
  `grant_type=authorization_code`, and the `code`, `code_verifier`,
  `redirect_uri`, and `client_id` values from the session and configuration.
- R-0RTW-DAQV — With a client secret configured, the body carries
  `client_secret` with that value; with none configured, no `client_secret`
  key is present at all.
- R-0U9P-4U89 — Each `--token-header` key/value is applied to the outgoing
  request as an HTTP header with that exact value, including an
  `Authorization` header, which reaches the server unaltered.
- R-0VHL-ILYY — A `--token-param` keyed `grant_type`, `code`, `code_verifier`,
  `redirect_uri`, `client_id`, or `client_secret` returns an error naming the
  key, and no HTTP request is issued.
- R-0WPH-WDPN — For a 2xx response, the returned bytes are byte-for-byte
  identical to the body the server sent: a body with non-alphabetical key
  order, irregular internal whitespace, and a trailing newline round-trips
  with all three preserved.
- R-0XXE-A5GC — A 400 response carrying an OAuth error body returns an error
  whose text includes the status and the provider's `error_description`, and
  returns zero bytes of payload.
- R-0Z5A-NX71 — A 2xx body that is not valid JSON is returned unchanged and
  without error, proving the payload is never parsed.
- R-10D7-1OXQ — A transport-level failure (server closes the connection
  without responding) returns an error and zero bytes.

These run against an `httptest.Server` acting as a **validating** token
endpoint: it asserts method, content type, and form fields, and fails the
request when they are wrong, so a malformed exchange cannot pass by being
politely accepted. Whether a real provider accepts the request is D7's claim.
