# Decision 5 — The browser launch seam

## Decision.

`internal/browser` opens a URL in the user's browser, and failing to do so
never fails a login.

```go
type Launcher interface {
    Open(url string) error
}

// New returns the platform launcher, or a no-op launcher
// on platforms with no supported open command.
func New() Launcher
```

**Platform files**, selected by build tag, each doing nothing but constructing
a command:

| platform | command |
|---|---|
| `linux` | `xdg-open <url>` |
| `darwin` | `open <url>` |
| everything else | no-op launcher, returns a "no supported launcher" error |

**Launch failure is always non-fatal.** The URL is printed to stderr
unconditionally, before any launch is attempted, so the user can act on it
whether or not a browser appears. `cmd/oauth-login` reports a failed launch on
stderr as a note and proceeds to wait for the callback. `--no-browser`
suppresses the launch entirely; the URL is still printed, so the two paths
differ only in whether a browser is spawned.

Windows is deliberately unimplemented. Adding it means shipping a code path
nobody here can exercise, and the no-op fallback already leaves such platforms
fully usable via the printed URL.

## Rejected.

- **Making a launch failure fatal.** The URL is already in the user's hands;
  aborting a login because a helper process was missing would be gratuitous.
- **Implementing Windows anyway.** Cheap to write, but unverifiable here, and
  the fallback path is already correct.
- **Detecting a headless environment and skipping the launch.** Guesswork
  (`DISPLAY`, `SSH_CONNECTION`, containers) that is wrong often enough to
  matter, replacing a harmless failed launch with a wrongly suppressed one.
  `--no-browser` states the intent explicitly instead.
- **Waiting on the spawned process.** `xdg-open` and `open` return once the
  browser is handed the URL; the login's completion signal is the callback,
  never the exit of the helper.

## Verification.

- R-1HFS-EHBG — On `linux`, `Open` invokes `xdg-open` with the URL as its sole
  argument, unmodified.
- R-1INO-S925 — On `darwin`, `Open` invokes `open` with the URL as its sole
  argument, unmodified.
- R-1JVL-60SU — When the launcher returns an error, the login continues to
  wait for the callback and a callback delivered afterwards still succeeds
  with exit 0; the failure is reported on stderr.
- R-1L3H-JSJJ — With `--no-browser`, the launcher is never invoked, and the
  authorize URL still appears on stderr.
- R-1MBD-XKA8 — On a platform with no supported open command, `New` returns a
  launcher that reports an error rather than panicking or executing anything,
  and the login proceeds exactly as in R-1JVL-60SU.

R-1HFS-EHBG and R-1INO-S925 assert the constructed command rather than
executing it, so they run without a desktop; the remaining three drive
`cmd/oauth-login` with a fake launcher, where the observable outcome is the
login's completion, not the launcher's return value.
