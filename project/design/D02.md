# Decision 2 — PKCE and authorize-URL construction

## Decision.

`internal/oauth` mints the PKCE pair and the `state` from an injected
`io.Reader`, then builds the authorize URL.

```go
type Client struct {
    AuthURL     string
    TokenURL    string
    ClientID    string
    RedirectURI string
    Scope       string        // omitted from the URL when empty
    Rand        io.Reader     // nil → crypto/rand.Reader
    // ...
}

// Session carries the per-login secrets from authorize through exchange.
type Session struct {
    State        string
    CodeVerifier string
}

func (c *Client) NewSession() (*Session, error)
func (c *Client) AuthorizeURL(s *Session, extra []Param) (string, error)

type Param struct{ Key, Value string }
```

**Verifier.** 64 random bytes from `Rand`, base64url-encoded without padding,
yielding 86 characters — inside RFC 7636's 43..128 bound and drawn only from
the unreserved set, so no further escaping is needed.

**State.** 32 random bytes (256 bits) from `Rand`, base64url without padding.

**Challenge.** `base64url_nopad(SHA256(ASCII(verifier)))`, sent with
`code_challenge_method=S256`. `plain` is never sent and cannot be selected.

**A short read from `Rand` is an error**, never a shorter secret: the
constructor uses `io.ReadFull`, and `NewSession` returns the error rather than
proceeding with partial entropy.

**Authorize URL.** Query parameters, all percent-encoded by `net/url`:
`response_type=code`, `client_id`, `redirect_uri`, `state`, `code_challenge`,
`code_challenge_method=S256`, and `scope` only when non-empty. `extra` params
from `--auth-param` are appended afterwards.

**Reserved-key rejection.** The six parameters above plus `scope` are owned by
this package. An `extra` param whose key matches any of them is an error
returned from `AuthorizeURL`, naming the offending key; it is never merged,
overridden, or silently dropped. `redirect_uri` is reserved along with the
rest, and its rejection message points at `--callback-host`, `--port`, and
`--callback-path` as the supported way to shape it.

## Rejected.

- **`crypto/rand` called directly.** Leaves the authorize URL
  non-deterministic, so tests could only assert shape ("43 characters,
  base64-ish"), which a broken-but-plausible implementation passes. Injection
  is what makes the RFC 7636 vector assertable.
- **An exported `Rand` field with no default.** A zero-value `Client` would
  then produce a predictable `state` in production. The constructor defaults
  nil to `crypto/rand.Reader` so the failure mode is impossible rather than
  merely discouraged.
- **`plain` as a fallback method.** Exists in the RFC for clients that cannot
  compute SHA-256. We can. Offering it only creates a way to downgrade.
- **Last-write-wins or first-write-wins for `--auth-param` collisions.** Both
  are quiet corruption: one silently discards the user's explicit flag, the
  other silently breaks PKCE or `state`.
- **A 43-character minimum-length verifier.** Legal, but there is no reason to
  sit at the floor of the permitted range when the cost of 64 bytes is nil.

## Verification.

- R-0EF0-5TL8 — A generated `code_verifier` is between 43 and 128 characters
  inclusive and contains only `ALPHA / DIGIT / "-" / "." / "_" / "~"`; a
  verifier containing any other character, or of length 42 or 129, fails.
- R-0FMW-JLBX — A generated `state` decodes to at least 16 bytes (128 bits) of
  entropy drawn from `Rand`; a 15-byte draw fails.
- R-0GUS-XD2M — The authorize URL carries `code_challenge_method=S256` exactly;
  no code path emits `plain` under any configuration.
- R-0I2P-B4TB — With `Rand` fixed so the verifier is RFC 7636 Appendix B's
  `dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk`, the emitted `code_challenge`
  equals `E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM` exactly, including the
  absence of `=` padding.
- R-0JAL-OWK0 — Two successive `NewSession` calls on one `Client` backed by
  `crypto/rand` produce different `state` values and different
  `code_verifier` values.
- R-0KII-2OAP — A `Client` constructed with a nil `Rand` draws from
  `crypto/rand.Reader` rather than panicking or emitting a zero-valued secret,
  and a `Rand` that returns fewer bytes than requested yields an error from
  `NewSession` rather than a short verifier.
- R-0LQE-GG1E — The authorize URL is the configured `AuthURL` carrying exactly
  `response_type=code`, `client_id`, `redirect_uri`, `state`,
  `code_challenge`, and `code_challenge_method`, each percent-encoded, with a
  `redirect_uri` containing reserved characters round-tripping intact.
- R-0MYA-U7S3 — `Scope` non-empty emits a `scope` parameter with that exact
  value; `Scope` empty emits no `scope` key at all, rather than `scope=`.
- R-0O67-7ZIS — Each `--auth-param` key/value appears in the URL exactly once,
  percent-encoded, and repeating the same key twice yields both occurrences.
- R-0PE3-LR9H — An `--auth-param` whose key is any of `response_type`,
  `client_id`, `redirect_uri`, `state`, `code_challenge`,
  `code_challenge_method`, or `scope` returns an error naming that key, and no
  URL is produced.

All of the above run against `internal/oauth` directly with a deterministic
`Rand`; none needs a network. The claim that a real provider accepts this URL
is verified by D7.
